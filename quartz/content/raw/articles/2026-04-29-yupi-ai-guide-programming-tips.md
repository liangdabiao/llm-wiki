# 鱼皮的 AI 指南 - 3、AI 编程技巧

## 前言

很多程序员对 AI 持怀疑态度，觉得 AI 只能做小网站小项目。

直到我纯用 AI、完全不写代码地做了一个几万行代码的大项目，才发现 AI 的进化远比想象更可怕。

核心问题：相同 AI 工具，为什么有人能做网站，有人只能做出来一堆 Bug？

---

## 一、优化 Prompt

Prompt 的质量直接决定了 AI 生成代码的准确性。

这也是为什么有 Prompt 工程的概念、出现了提示词工程师这个岗位。

### 基本 Prompt 优化技巧

1. 明确 AI 的角色定位
2. 提供具体的细节要求
3. 拆解任务
4. 提供示例

参考学习：智谱 AI 开放平台文档 https://open.bigmodel.cn/dev/guidelines/LanguageModels

### 进阶技巧：AI 生成高质量 Prompt

把自己手写的 Prompt 扔给其他 AI，让它帮你生成一段质量更高的、结构化的需求文档，再喂给 Cursor 去生成网站，效果会更好。

**完整需求文档通常包括**：
- 明确项目背景和目标
- 详细列出功能要求和技术栈
- 指定代码风格和架构模式
- 提供示例和参考资料
- 明确限制条件和边界场景

## 二、复杂项目的生成技巧

复杂项目不可能一步到位，采用 **分步迭代** 策略：

1. **生成基础框架**：Cursor Agent 模式 + 高级推理模型生成基础项目框架，确保能跑起来就行，暂时不管功能好不好用
2. **逐步实现核心功能**：划分模块和功能点，依次向 AI 提问，逐步实现核心功能并验证可用
3. **优化实现细节**：在不影响功能的前提下，优化实现细节

## 三、常见问题解决方法

常见问题：
- AI 输出驴头不对马嘴
- 生成新功能后，原来的功能不能用了
- 改了一个 Bug，又来 3 个 Bug

### 1. 项目模块化

AI 的"脑容量"不大，上下文有限，项目信息量增大后 AI 可能忘记之前的信息，导致生成错误代码。

**解决方法**：把项目功能尽可能隔离，将大项目分割成多个小项目。AI 生成某个功能时只需关注一小部分上下文，结果更精准。

**例子**：开发电商系统时，商品管理模块独立出来，生成添加商品功能时，只提供商品表字段设计和业务逻辑规则，不需要提供支付结算、用户会员等关联不大的信息。

### 2. 限定修改范围

AI 生成代码不可控，经常改 A 功能的同时把 B 功能也顺带修改了。

**解决方法**：在提示词中明确限定修改范围。

**示例**：
```markdown
仅修改 services/order.java 中的 CreateOrder 方法：
1. 添加分布式锁防止超卖  
2. 保持现有日志格式  
3. 不要改动其他文件
```

### 3. 抽象和复用

AI 生成代码有时很死板，两个布局一模一样的页面，AI 可能复制一遍代码，不利于维护。

**解决方法**：适当提醒 AI：请帮我抽象 XX 页面、XX 代码为可复用的组件。

好处：
- 减少整个项目代码量
- 减轻 AI 上下文记忆负担
- 生成结果更准确

### 4. 版本控制

利用 Git 版本控制工具管理代码：
- 每正确生成一个功能后提交一次版本
- 每次 AI 生成新代码后人工对比改动文件
- 出问题快速还原到之前版本，防止代码丢失

## 四、其他小技巧

1. **注意操作系统**：AI 更喜欢 Linux 或 Mac 系统终端，执行命令能力更强。Windows 用户建议安装 WSL，或在对话中明确告诉 AI 使用 Windows 命令。

2. **人工控制**：AI 缺乏关键上下文信息或自身能力不足时，会陷入循环。此时人工介入：手动指定上下文、更换 Prompt 引导 AI。

3. **多元 AI 协作**：不同大模型擅长不同任务。单一模型无法完成时，利用其他大模型生成"教 AI 做事的方法和指令"，增加解决问题可能性。
