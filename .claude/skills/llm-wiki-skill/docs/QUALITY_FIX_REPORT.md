# llm-wiki skill 质量修复报告

> 日期：2026-04-29
> 修复目标：系统性解决知识库质量问题，防止未来再次发生

---

## 一、发现的问题汇总

### 问题 1: 实体页 sources 字段普遍为空

**严重程度：🔴 高**

**现象：**
- 12 个实体页的 frontmatter 中 `sources: []` 是空数组
- 导致知识来源无法追溯
- 违反质量保证清单第 776 条

**根因分析：**
1. 模板中有 `sources: [{{FIRST_SOURCE}}]` 占位符，但执行时没有正确替换
2. 单个 ingest 工作流没有后置检查机制
3. 质量清单只是文字要求，没有强制执行

---

### 问题 2: 素材摘要缺少必填节（实操内容 + 原文摘录）

**严重程度：🔴 高**

**现象：**
- 8 篇素材摘要缺少"实操内容保留"节
- 7 篇素材摘要缺少"原文精彩摘录"节
- 实际信息量比预期少 30-40%

**根因分析：**
1. 模板中有要求，但执行时 AI 容易跳过
2. 没有自动化检查脚本强制要求填写
3. 批量消化时为了速度牺牲内容完整度

---

### 问题 3: 文件名与链接名不一致导致断链

**严重程度：🟡 中**

**现象：**
- 文件名：`DeepSeek-Agent.md`
- 链接名：`[[DeepSeek Agent]]`（空格 vs 横杠）
- 导致点击链接找不到文件

**根因分析：**
1. 没有"先确定文件名，再写链接"的严格顺序
2. 创建文件和写链接是两个独立步骤，容易不一致
3. 没有自动检查机制

---

### 问题 4: 批量消化后置检查只在批量时运行，单个 ingest 不运行

**严重程度：🟡 中**

**现象：**
- SKILL.md 中只有 batch-ingest 工作流有后置检查
- 单个素材 ingest 完成后不做任何质量检查
- 问题累积到很多时才发现，修复成本高

**根因分析：**
1. 最初设计只考虑了批量场景
2. 单个 ingest 场景遗漏了质量检查
3. 没有质量门禁（quality gate）概念

---

### 问题 5: 实体页内容深度不达标

**严重程度：🟡 中**

**现象：**
- LangChain4j.md 只有 1393 字符（要求最低 1500）
- 部分实体页只有骨架，没有实质内容

**根因分析：**
1. 为了速度只写了框架，没有填充细节
2. 字数检查只是建议，不是强制执行
3. 缺少"如果不达标怎么办"的明确指引

---

## 二、已实施的修复措施

### 措施 1: 新增自动化质量检查脚本

**文件：** `scripts/lint-wiki.sh`

**功能：**
```bash
# 完整检查
bash lint-wiki.sh <知识库路径> all

# 单项检查
bash lint-wiki.sh <知识库路径> links     # 断链检查
bash lint-wiki.sh <知识库路径> sources   # sources 字段检查
bash lint-wiki.sh <知识库路径> content   # 实操内容和摘录检查
bash lint-wiki.sh <知识库路径> depth     # 内容深度检查
```

**检查项：**
1. ✅ 链接完整性检查（断链检测）
2. ✅ sources 字段完整性检查（空数组、占位符）
3. ✅ 内容必填节检查（实操内容、原文摘录）
4. ✅ 内容深度检查（最低 1500 字符）
5. ✅ index.md 完整性检查

**输出：**
- 通过 / 警告 / 错误 三级分级
- 红色高亮错误，黄色高亮警告
- 错误数量 > 0 时退出码为 1，可被 CI/CD 捕获

---

### 措施 2: SKILL.md 强化执行要求

**修改 ingest 工作流：**

1. 在流程开始处增加醒目的警告：
   ```
   ⚠️ 强制执行：所有步骤完成后必须运行质量检查脚本，不达标不能结束！
   ```

2. 在原步骤 9 和 10 之间新增步骤 10（强制执行质量检查）：
   ```bash
   bash ${SKILL_DIR}/scripts/lint-wiki.sh "${WIKI_PATH}" links sources content
   ```
   - 如果有错误 → 立即修复
   - 如果有警告 → 能修复的立即修复，不能修复的记录到 log.md

**修改位置：** SKILL.md 第 223 行、第 354 行附近

---

### 措施 3: 模板增加醒目必填提示

**entity-template.md、source-template.md、topic-template.md** 均在顶部增加：
```markdown
<!-- ⚠️ 重要：本模板中的所有占位符必须用实际内容替换，禁止留空！ -->
```

---

### 措施 4: 实体页"文件名=链接名"一致性规则强化

在 SKILL.md 质量保证清单中增加执行级别的要求：

```
生成实体页时的铁律（强制执行）：
1. 先确定实体的标准名称（例如：DeepSeek Agent）
2. 用这个名称创建文件：`wiki/entities/DeepSeek Agent.md`
3. 在所有其他页面引用时，必须使用完全相同的名称：`[[DeepSeek Agent]]`
4. 禁止使用不同的名称（如 DeepSeek-Agent、DeepSeek agent 等）
```

## 三、质量保证新机制（防止再次发生）

### 三层质量防护

```
第一层：模板约束
   ↓
第二层：执行时自检
   ↓
第三层：自动化脚本验证
```

#### 第一层：模板约束
- 每个模板顶部都有醒目的必填提示
- 用 `<!-- ⚠️ 重要：... -->` 注释标记
- 禁止留空，禁止用占位符

#### 第二层：执行时自检
- 每个 ingest 工作流末尾必须运行质量检查清单
- 不通过的立即修复，不延后
- 错误记录到操作日志

#### 第三层：自动化脚本验证
- lint-wiki.sh 自动检查常见问题
- 错误 > 0 时流程终止，强制修复
- 可集成到 CI/CD 流程中每日运行

---

### 质量门禁规则

| 级别 | 标准 | 处理方式 |
|------|------|---------|
| **错误（Error）** | sources 为空、断链、文件名与链接名不一致 | 必须立即修复，流程不允许结束 |
| **警告（Warning）** | 内容过短、摘要缺少可选节、统计数字不一致 | 尽可能修复，如时间紧迫记录到日志，下次处理 |
| **建议（Advice）** | 可以优化但不影响功能 | 记录，后续优化 |

## 四、最佳实践更新

### 实体页创建 SOP（标准作业流程）

1. **确定标准名称**
   - 概念的标准叫法是什么？
   - 有没有简称和全称？
   - 写下来，这就是文件名 = 链接名

2. **创建文件**
   - 用标准名称创建：`wiki/entities/[标准名称].md`
   - 复制模板内容

3. **填充内容**
   - frontmatter 的 sources 字段立即填写
   - 正文所有节一次性填满，不留"待补充"
   - 至少 1500 字符

4. **添加链接**
   - 在 index.md 中添加条目
   - 在相关的主题页、其他实体页中添加 `[[标准名称]]` 链接
   - **绝对不要修改名称，必须与文件名完全一致**

5. **验证**
   - 运行 `bash lint-wiki.sh <路径> links sources`
   - 确认没有错误

---

### 素材摘要创建 SOP

1. **保存原始素材**到 raw/ 目录
2. **提取核心观点**（至少 3 个，每个要有数据/案例）
3. **复制实操内容**（代码、命令、步骤、模板）
4. **摘录原文精彩段落**（至少 2 段）
5. **填写 source_path** 指向 raw/ 目录中的文件
6. **运行检查** `bash lint-wiki.sh <路径> content`

## 七、结论

本次修复从**机制层面**解决了 llm-wiki skill 的质量问题，而不仅仅是修复了现有知识库的个别问题。

核心改进：
1. ✅ **从"软要求"到"硬约束"**：质量清单不再只是文字，而是有脚本强制执行
2. ✅ **从"批量检查"到"逐个检查"**：每个 ingest 都检查，不累积问题
3. ✅ **从"事后修复"到"事前预防"**：模板和流程强化，从源头减少错误
4. ✅ **从"人工检查"到"自动化检查"**：脚本自动检查，减少人为疏漏

建议后续每次使用 llm-wiki skill 时都运行 lint-wiki.sh 作为质量门禁。
